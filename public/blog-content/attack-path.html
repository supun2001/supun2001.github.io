Cloud Privilege Escalation: The “Identity” Attack Path

In 2025, cloud breaches rarely start with a zero-day.

Instead, they start with something much quieter: a misconfigured identity.

In modern cloud environments, identity is the perimeter. If an attacker can abuse permissions, they don’t need malware, exploits, or infrastructure bugs.

This article breaks down how cloud privilege escalation actually happens in AWS and Azure, and why identity misconfigurations are the fastest path to admin.

The Shift: From Network Attacks to Identity Attacks

Traditional security focused on:

  • Firewalls
  • Network segmentation
  • Exposed services

Cloud-native environments flipped the model.

Today, access is controlled by:

  • IAM roles
  • Policies
  • Service permissions
  • Trust relationships

If identity is misconfigured, everything behind it is exposed, even if the network is “secure.”

What Is Cloud Privilege Escalation?

Cloud privilege escalation occurs when a user or role:

  • Starts with limited permissions
  • Finds unintended access paths
  • Chains permissions together
  • Ends with elevated privileges

The key insight:

No single permission looks dangerous on its own.

The danger appears when permissions are combined.

Common Entry Point: “Read-Only” Access

Many cloud incidents begin with what looks harmless:

  • A read-only IAM user
  • A compromised CI/CD token
  • An exposed cloud credential
  • A leaked access key in a repo

Teams often assume:

“It’s only read access, there’s nothing they can do.”

That assumption is wrong.

Why Read Permissions Are Dangerous

Read access enables reconnaissance.

With enough visibility, an attacker can:

  • Map IAM roles and policies
  • Identify overly permissive roles
  • Discover trust relationships
  • Find services that can assume other roles

This is where cloud attacks differ from traditional pentesting.

The attack surface is logical, not technical.

Methodology: Mapping IAM Like an Attacker

A real-world cloud privilege escalation workflow follows a predictable pattern.

Step 1: Enumerate Identity Permissions

The goal is to answer:

  • Who am I?
  • What can I see?
  • What services can I interact with?

Even limited permissions often allow attackers to list roles, policies, and attached permissions.

Step 2: Identify Privilege Escalation Paths

Attackers look for permissions that enable:

  • Role assumption
  • Policy modification
  • Credential creation
  • Service impersonation

Individually, these permissions may seem acceptable.

Combined, they become an escalation chain.

Step 3: Abuse Trust Relationships

Cloud environments rely heavily on trust:

  • Services trusting roles
  • Roles trusting other roles
  • Cross-account access

If a role trusts something it shouldn’t, attackers can pivot without exploiting a single vulnerability.

Scenario: From Read-Only to Administrator

A common escalation path looks like this:

  • Start with read-only IAM access
  • Discover a role with broader permissions
  • Identify a service allowed to assume that role
  • Trigger the service in an unintended way
  • Gain access under a more powerful identity

No exploits. No malware. Just permissions behaving exactly as configured.

Tooling: Pacu (AWS Exploitation Framework)

Manually mapping cloud permissions is slow and error-prone.

This is where tools like Pacu come in.

Pacu helps security teams and attackers alike by:

  • Enumerating IAM permissions
  • Identifying known escalation paths
  • Mapping role trust relationships
  • Highlighting risky configurations

The real value of Pacu is not automation, it’s visibility.

Why These Attacks Are Hard to Detect

Identity-based attacks often:

  • Use legitimate APIs
  • Appear as normal cloud activity
  • Generate no obvious alerts
  • Blend into daily operations

From a logging perspective, nothing looks “exploited.”

Everything is technically working as designed.

What Blue Teams Get Wrong

Common defensive mistakes include:

  • Overly broad IAM roles
  • Unused permissions left in place
  • Unreviewed trust relationships
  • No continuous IAM auditing

Security teams often focus on:

“Who can access what?”

But attackers focus on:

“Who can become who?”

Defensive Takeaways

To reduce identity-based privilege escalation:

  • Apply least privilege aggressively
  • Audit IAM roles continuously
  • Review trust relationships regularly
  • Monitor role assumption events
  • Remove unused permissions

In the cloud, identity sprawl is attack surface.

Final Thoughts

Cloud security in 2025 is not about patching servers.

It’s about controlling identity.

If attackers can see your permissions, they can often see their path to admin.

No zero-day required.
Just one misconfigured role.

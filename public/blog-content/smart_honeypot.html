Why Blocking Attackers Is No Longer Enough in 2025

For a long time, cybersecurity focused on one simple idea: block the attacker as fast as possible.

Firewalls, EDRs, and SIEM platforms are all built around this mindset, detect suspicious activity, raise an alert, and shut it down.

That approach still matters. But in 2025, the reality has changed.

Modern attackers are:

  • Highly automated
  • Patient and persistent
  • AI-assisted
  • Scanning thousands of systems simultaneously

Blocking them is necessary, but on its own, it’s no longer enough.

This is where Deception Engineering becomes powerful.

What Is Deception Engineering?

Deception Engineering is the practice of intentionally deploying fake assets inside your environment to lure attackers.

The goal isn’t to stop them immediately, it’s to:

  • Detect intrusions early
  • Waste attacker time and effort
  • Collect high-quality threat intelligence
  • Protect real systems from being touched

Instead of asking: “How do we block everything?”

Deception asks a better question: “How do we make attackers expose themselves?”

Why Deception Is a Major Trend in 2025

In modern environments:

  • Attackers assume they are already inside
  • Zero Trust limits movement but doesn’t stop initial access
  • Detection speed matters more than prevention alone

Deception works because:

  • Legitimate users never interact with decoys
  • Any interaction is suspicious by default
  • False positives are extremely rare

Simply put: If someone touches your honeypot, they’re not a real user.

Step 1: Deploying Canarytokens (The Simplest Honeypot)

What Are Canarytokens?

Canarytokens are unique digital tripwires that alert you the moment they are accessed.

They can be embedded in:

  • Documents
  • AWS access keys
  • DNS records
  • URLs

If someone opens or uses them, you receive an alert immediately.

Canarytoken in a Word Document

This is one of the easiest ways to detect:

  • Internal snooping
  • Post-compromise exploration
  • Attackers searching for sensitive data

How I set it up:

  1. Create a Microsoft Word Canarytoken
  2. Download the generated document
  3. Rename it to something realistic and tempting

Example filenames:

  • HR_Salaries_2025.docx
  • Cloud_Credentials.docx
  • Finance_Backup_Q1.docx

Where to Place the Canarytoken Document

The file should be placed where attackers look, but users don’t.

Ideal locations include:

  • Old or archived shared folders
  • Legacy infrastructure directories
  • Backup or migration-related locations

Example realistic path: 

\\backup-nas\legacy-systems\HR_Salaries_2025.docx

This works because:

  • Attackers actively enumerate shared folders
  • “Legacy” implies valuable but outdated data
  • Legitimate users rarely access these paths

Rule of thumb: If a real user might open the file, it’s in the wrong place.

Canarytoken as a Fake AWS Key

Attackers aggressively hunt for cloud credentials.

Canarytokens allows you to generate a fake AWS access key that alerts you the moment it’s used.

This is one of the highest-signal deception techniques available in 2025.

Step 2: Deploying OpenCanary (Your First “Smart” Honeypot)

What Is OpenCanary?

OpenCanary is an open-source, low-interaction honeypot that emulates vulnerable services.

It can pretend to be:

  • SSH server
  • FTP service
  • SMB / NAS device
  • HTTP service

Every interaction is logged.

Installing OpenCanary

I deployed this on a small Linux VM (Ubuntu 24.04 works perfectly): 

sudo apt update
sudo apt install python3-pip -y
pip3 install opencanary
opencanaryd --copyconfig

Full installation details are available in the official OpenCanary GitHub repository .

Configuring OpenCanary as a Vulnerable NAS

Attackers love NAS devices, they often store backups and sensitive data.

I configured OpenCanary to appear as a NAS by enabling SMB: 

"smb.enabled": true,
"smb.port": 445,
"smb.domain": "WORKGROUP",
"smb.server_name": "backup-nas"

On the network, this appears as: backup-nas

No real files. No real risk.

Starting the Honeypot

opencanaryd --start

At this point, your fake NAS is live.

When the Attacker Bites - Reading the Logs

Honeypot logs are different from normal logs.

There’s only one real question: Why is someone touching this? 

{
  "dst_port": 445,
  "service": "smb",
  "src_host": "185.xxx.xxx.xxx",
  "utc_time": "2025-03-18T11:42:21Z"
}

In a real SOC, this alert is treated as high confidence by default.

Important Note on Responsible Use

Deception tools should only be deployed in environments you own or are authorised to test.

They are designed for detection and visibility, not surveillance or entrapment.

What Not to Do (Very Important)

Deception tools are powerful, but they must be used responsibly.

Do NOT:

  • Capture or store real credentials
  • Imitate real companies or brands
  • Deploy honeypots on school or work networks without permission
  • Add exploits, malware, or active attack code

Your deception lab should always be:

  • Detection-only
  • Read-only
  • Passive

When deployed correctly, deception provides early detection with minimal risk, making it one of the most effective defensive techniques in 2025.

Final Thoughts

In 2025, security isn’t just about stronger walls.

It’s about smarter traps.

Prevention tries to stop attackers, deception lets them identify themselves.
